Diagnosis hypotheses for S20 P0 (in order of expected cost vs likelihood):
1. **Console errors not yet captured** — F12 → Console on BOTH the host
   browser AND the client browser during the connect attempt. Look for
   WebSocket close codes (`1006` = abnormal close, `1011` = server error),
   `wss://` failures, `ICE candidate gathering failed`, peer connection state
   transitions (`new → connecting → failed`). The error text usually names
   the failure layer directly.
2. **Trystero 0.24 API wrapper drift** — `src/net/transport.ts:54-56` casts
   `room.makeAction('msg')` through `unknown` to bypass type checking; 0.24
   may have changed the signature or moved `makeAction` to a different object.
   Pull the 0.24 `Room` type from `@trystero-p2p/core` and validate the call
   site against actual API.
3. **ICE/TURN missing for symmetric NAT** — Trystero 0.24 uses Google STUN
   defaults. If user OR brother is on symmetric NAT (mobile hotspot, corporate
   network, some ISP CGNATs), STUN-only fails. Need TURN server. Free options:
   Cloudflare TURN (open beta, free), Twilio (1 GB/mo free), self-host coturn.
   Pass via `rtcConfig: { iceServers: [...] }` to `joinRoom`.
4. **Trystero 0.20 → 0.24 broke peer handshake itself** — A/B test by
   downgrading: `npm install trystero@0.20.0` → rebuild → retest. If 0.20
   works on same network, the bump is at fault and we either stay on 0.20
   or open a Trystero issue. If 0.20 also fails, network/NAT is the layer.
5. **Strategy swap** — if Nostr 0.24 is genuinely broken, switch to
   `trystero/mqtt` (free public MQTT brokers, well-supported in 0.24) or
   `trystero/torrent` (Council R1 originally REJECTed for rate limits but
   that was 0.20-era; 0.24 may have improved). LOCKED §13.1 would amend
   to v5 with the new strategy.
