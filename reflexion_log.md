## 2026-05-17 — Session 35 P0 (1v1 join bootstrap deadlock — bug existed since S15 add497f, ~20 sessions; commit 1be5fce)

- S35-P0 #integration-blind-spot-since-S15-because-unit-tests-bypass-main-ts-inline-gates: 1v1 join bootstrap deadlock at main.ts:765 client-interpolation gate existed in production since S15 commit add497f (introduction of 1v1 networking). 20 sessions of subsequent 1v1 work (S20 error plumbing, S22 P3 godly broadcast, S31 P0-3 effects mirror, S33 P1-11 creatureId additivity, S34 multiple) were validated only via vitest unit tests that exercise ClientSync.receive + interpolateInto directly — bypassing the main.ts inline render-loop gate that prevents interpolateInto from being called at all on joiner. The boot-snapshot's recurring "1v1 brother retest pending" carry item across S31/S32/S33/S34 is the symptom: cross-network 2-peer playtest kept being deferred, likely because it was IMPOSSIBLE — the joiner never left LOBBY, so post-PLAYING bug retests (effects mirror, creature additivity) couldn't even be attempted. **Lesson: any code that lives inside main.ts inline ticker callbacks (or other Pixi/DOM-gated code paths) has ZERO unit-test coverage by default. The vitest pattern of "construct a World, call reducer/sync/lerp directly" mocks AROUND the main.ts wire-up. For state-machine wiring like the join/host handshake, the gates BETWEEN reducers and renderers are the highest-leverage bug surface AND the hardest to test. Mitigation options: (a) extract main.ts inline gate logic into pure helpers, (b) Playwright/jsdom integration test for the LOBBY→PLAYING transition, (c) explicit playtest gate in PDR execution order. Option (c) is what S35 P0 used — cheap but human-loop-dependent.**

- S35-P0 #asymmetric-host-joiner-setup-needs-symmetry-check-on-every-1v1-relevant-world-field: Host gets world.gameMode='1v1' via applyStartGame reducer dispatch at onBeginMatch. Joiner is supposed to mirror via NETSNAPSHOT apply, BUT the apply path is gated on gameMode='1v1' — Catch-22. Root cause is that onJoinAttempt sets isHost=false but not gameMode='1v1', breaking the bootstrap-from-snapshot pattern. **Lesson: any new world.* field that host sets via reducer should be checked for joiner-side parity in onJoinAttempt. Specifically: any field that is later used as a GATE for the snapshot-apply path must be set on joiner BEFORE first snapshot. Pattern carry-forward for S35+ Anvil + future networked features: when adding gameMode-style flags, audit whether they appear in render-loop gates that themselves gate the snapshot-apply path. The bootstrap-from-snapshot pattern is only safe for fields that AREN'T gates on the path that applies the snapshot.**

- S35-P0 #prime-audit-all-gates-sweep-before-state-mutation-in-entry-point-handler: Before adding `world.gameMode = '1v1';` to onJoinAttempt, PRIME-AUDIT swept all 11 `gameMode === '1v1'` references across main.ts (6 sites) + state/ (4 sites) + render/ (1 site) for misfire risk in the post-fix interim state (gameMode='1v1' + gameState='LOBBY' on joiner). - S35-P0 #bootstrap-gate-catch-22-pattern-codified: A "bootstrap gate Catch-22" occurs when a gate-controlled state mutation is itself gated on the result of that mutation. main.ts:765 gates clientSync.interpolateInto on gameMode='1v1', AND interpolateInto is the only path that sets gameMode='1v1' (via applyNetSnapshot via applySnapshotCore at save.ts:313). Without an external bootstrap (setting gameMode='1v1' OUTSIDE the gate's control), the system is permanently stuck. **Lesson: any pattern like `if (world.X === V) { ... mutates world.X to V ... }` is a deadlock unless world.X is set to V somewhere else first. Audit checklist for future gate-and-mutate paths: (1) does the gate condition appear as the OUTPUT of the gated mutation? (2) is there an INDEPENDENT path to set the gate-condition? If (1) is yes AND (2) is no, it's a deadlock. Naming: 'bootstrap gate Catch-22'.**

- S35-P0 #user-driven-playtest-as-final-validation-gate-when-unit-tests-cannot-reach-the-failure-site: Trystero/Nostr P2P needs 2 separate browser instances (different signal IDs) — single-browser preview workflow cannot validate.
