- S39 #object-fit-contain-letterbox-coordinate-mapping-trap: BUG-B was caused by `controls.updateCursor` using non-uniform `sx = CANVAS_W/rect.width`, `sy = CANVAS_H/rect.height`. The canvas is rendered with `object-fit: contain` (Pixi default) — under any viewport aspect ≠ canvas aspect, content is letterboxed inside the CSS box and `getBoundingClientRect()` returns the full box including bars. The non-uniform formula was correct ONLY at matched aspect; at any other aspect the cursor mapping diverged from visible canvas content by up to the letterbox-bar size. Bug is invisible at visual center (where both formulas agree on canvas-center) and grows linearly to ±72 CSS-px at the visible canvas edges (verified empirically at 1280×900 viewport: pre-fix drifts -90/-72/-54/0/+50/+72/+90 across CSS-y axis; post-fix 0 everywhere). The same buggy math was ALSO present in `mapCanvasRectToPage` (latent in HTML input positioning). Fix: shared pure helpers `cssToCanvasCoords` + `fitCanvasIntoRect` in lobbyScreen.ts compute the letterbox-aware uniform scale + offset; both call sites switch to the helpers. **Lesson: any code that maps between CSS-px and canvas-units MUST use letterbox-aware math when the canvas is rendered with object-fit:contain (or its `cover` / `fill` cousins). The "simple ratio of widths" formula is a trap — it gives correct answers at the visual center (because center is invariant under any aspect transform) and wrong answers everywhere else. Pattern carry-forward: codify a single canonical helper for canvas↔CSS conversions in this codebase; any new call site MUST import from it instead of inlining the math.**

- S39 #wire-protocol-handler-idempotence-via-state-gate: The peer's START_GAME_SIGNAL handler at main.ts:onJoinAttempt is guarded by `world.gameState === 'LOBBY'`. Cost: 6 chars (`world.gameState === 'LOBBY'`). Benefit: handler is idempotent against the entire class of "message-arrives-after-state-has-already-progressed" failure modes, without requiring sequence numbers or de-dup logic. **Lesson: every wire handler whose action would RE-INITIALIZE state (vs. INCREMENTALLY UPDATE state) should gate on the current state. Snapshot handlers update incrementally so they don't need this; control signals (START_GAME_SIGNAL, END_GAME, RETURN_TO_TITLE) reset blocks of state so they do. Pattern carry-forward: any future control signal that triggers an `apply*` reducer with reset semantics should follow the same state-gate pattern.**
