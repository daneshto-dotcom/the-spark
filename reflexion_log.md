- S40 #compound-deploy-failures-need-empirical-layer-strip-not-conjecture-of-single-cause: S40 found **3 distinct compounding failure modes**, each independently sufficient to block deploys: (a) phantom github-pages deployment locking env concurrency, (b) cascade-cancel via rapid session-bookkeeping pushes, (c) billing-blocked runner allocation. Layers don't compose by inference: a workflow_run can sit pending forever for reasons that have nothing to do with the deploy.yml content. **Lesson: when production is unexpectedly stale, do NOT settle on the first plausible explanation. Strip the failure layers empirically using the API endpoint at each layer of the stack. The github-pages deploy pipeline has at least 4 separate state machines (deployments, workflow_runs, jobs, check-runs/annotations) and each can be stuck independently. Pattern carry-forward: codify a deploy-troubleshooting runbook: (1) last successful deployment age; (2) any non-terminal deployment records?; (3) any non-terminal workflow runs?; (4) jobs that started_at but never got a runner?; (5) annotations on check-runs of failed jobs. Each step a single `gh api` call; together they triangulate the failure layer in seconds.**
