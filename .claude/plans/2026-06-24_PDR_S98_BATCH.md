## P1 — Fix the E2E CI lane

**OBJECTIVE.** The gating `e2e` job goes reliably GREEN; the flaky real-WebRTC suite can never cancel
or red it. Stops the failure emails.

**ROOT CAUSE (empirical).** One workflow runs Playwright: `.github/workflows/e2e.yml`, job `e2e`,
`timeout-minutes: 15`. The gating/non-gating split is **two steps in one job**: a gating step
`npm run e2e -- --grep-invert "@quarantine-flaky"` and a non-gating step (`if: always()`,
`continue-on-error: true`) `--grep "@quarantine-flaky" ...`. Because both run in the **same job**, the
quarantine step's slow real-WebRTC retries (each S82=120s / S63=90s test × `retries: 2`) push the
single job past the 15m cap → the **whole job is cancelled**, which moots the already-passed gating
step. That cancellation is what emails the user.

**HANDOFF CORRECTION (PRIME-AUDIT row).** The handoff said "quarantine 4 flaky gating tests
(S62/S63 FFA, S82 reconnect)." Empirically:
- **S62 3-player FFA** — `e2e/smoke.spec.ts:666` — **UNTAGGED, still GATING** → needs a new tag.
- **S63 4-player FFA** — `e2e/nplayer.spec.ts:67` — **already `@quarantine-flaky`**.
- **S82 reconnect** — `e2e/reconnect.spec.ts:19` — **already `@quarantine-flaky`**.
- **S70 presence** — `e2e/nplayer.spec.ts:291` — **already `@quarantine-flaky`** (4th real-WebRTC).
So the real fix is the **job split**; the only new per-test tag is **S62**.

**APPROACH.**
1. **Split, don't relocate.** Keep `.github/workflows/e2e.yml` (preserve the required-check name
   `e2e` for branch protection). Move the quarantine step into its **own job** `e2e-quarantine`
   in the same file, `needs: []` (runs in parallel, cannot delay/cancel `e2e`). It keeps
   `continue-on-error`-equivalent semantics by being non-required + `if: always()` internally and
   `--reporter=list --output=test-results-quarantine` (so it never overwrites the gating HTML report).
   Give it its own adequate `timeout-minutes` (e.g. 20) so its retries don't matter — it never gates.
2. The gating job `e2e` runs **only** `--grep-invert "@quarantine-flaky"` and keeps `timeout-minutes: 15`
   (it has plenty of headroom once the slow WebRTC suite is gone).
3. **Tag S62**: append ` @quarantine-flaky` to the `test.describe(...)` title at `smoke.spec.ts:666`,
   with a preceding comment block stating the flake class (`net::ERR_ADDRESS_UNREACHABLE`, CI sandbox
   can't hold 3-peer P2P; deterministic transport rejected per LOCKED_DECISIONS) — mirror the existing
   idiom at `nplayer.spec.ts:61-67`.
4. **DRY (optional, low-risk):** add `e2e:gating` / `e2e:quarantine` npm scripts wrapping the grep
   flags, and call them from the YAML.

**STATE-DISCOVERY (must run before/at execution).** `gh api` the repo's branch protection /
required status checks to confirm the required check is the **job name** `e2e` (not the workflow
name). If a different name is required, adjust so the gating check name is preserved. If `gh` can't
read protection (perms), keep the gating job name `e2e` unchanged (safest) and note the assumption.

**OUT OF SCOPE.** Adding `needs: [e2e]` to `deploy.yml` (E2E still won't gate deploy — separate
decision, the S46 migration note). De-flaking WebRTC-in-CI itself (environmental; quarantine is the
LOCKED policy).

**TESTING.** `actionlint`/YAML sanity; confirm `npm run e2e:gating` and `e2e:quarantine` resolve the
right test sets locally (`--list`); push and confirm on GitHub: `e2e` job GREEN, `e2e-quarantine` job
runs independently (red/yellow there does NOT email or gate). vitest/tsc unaffected.

**ROLLBACK.** Revert the single workflow + spec-tag commit; the prior single-job config returns.

## P2 — Combo Order-Symmetry (Option B)

**OBJECTIVE.** Connecting the same two shape types yields the same magic regardless of carry order,
for the **8 one-way magic pairs**; the intentional **Triangle↔Circle = Wheel/Star** duality is
preserved; HUD stays **N/14**; persistence and the discovery count don't break.

**THE 8 PAIRS.** Dot↔Line (Filament), Dot↔Square (Anchor), Dot↔Spiral (Vortex), Line↔Triangle
(Bracket), Line↔Circle (Spindle), Line↔Spiral (Whip), Triangle↔Spiral (Warped Anchor),
Square↔Circle (Capsule). Each: the reverse order should yield the **same** magic (user-confirmed).

**APPROACH — canonicalize the TABLE, not the key.**
- In `src/combos.ts`, add the **8 reverse rows** to the `MAGICAL[]` literal (e.g. `[Spiral, Triangle,
  <WarpedAnchor outcome>]`), so `lookupCombo` returns the same `ComboOutcome` in both orders. Leave
  `comboKey()` **unchanged** (no sort) → no key-format change → no localStorage / save / replay
  migration, no parse breakage. Leave `Triangle→Circle`/`Circle→Triangle` as the two distinct rows.
- The functional-fill loop (auto-generates `Bond_a_b` placeholders for unset keys) now fills 8 fewer
  cells; `TABLE.size` stays **36** (still 6×6). Update the `TABLE.size === 36` invariant comment if
  needed (count is unchanged — magic rows 14→22, functional fill 22→14).  **Verify the count math in
  code, not just here.**
- **Keep N/14 unordered.** `MAGIC_COMBO_KEYS` / the codex catalog must show **14 named magics**, not
  22 keys. Implement by deriving the catalog/count from **distinct `resultName`** (or a canonical
  representative key per magic), and in `detectComboDiscoveries`, record a discovered magic under its
  **canonical key** so forward+reverse unlock the same tile and never double-count. Existing persisted
  `spark:combos:discovered:v1` keys (the 14 forward keys) remain valid — no migration.
- **Behavior helpers** (`isVortexCombo`, `isFilamentCombo`, etc.) key off `resultName` → they follow
  the new rows automatically, no change.

**REBALANCE — derive, don't assume (this is the nuanced call).**
- The LOCKED ~157.5s anchor tracks the **canonical (optimal) build**, which already places in magic
  (forward) order → it already earns every magic. Symmetry adds **no new magics and no higher
  multipliers**; it only lets the 8 one-ways fire in reverse too → it raises the **floor** for sloppy
  play, not the **ceiling** for optimal play. ⇒ optimal-build income (and thus the 157.5s anchor) is
  **unchanged**.
- **Recommendation:** ship Option B with **PHASE_1_WIN_SCORE unchanged (630)**. A blanket inflation
  would *over-correct* — making optimal-play matches longer than 157.5s and penalizing skill. The
  rebalance the original PDR feared assumes average-occurrence doubling drives the anchor; it does not.
- **Safety valve:** keep a ready-to-apply micro-rebalance (raise `PHASE_1_WIN_SCORE` + `SCORE_TIER_STEP`
  in lockstep preserving the `/3` invariant) **gated on the user's playtest** — "do casual matches feel
  too fast now?" If yes, apply; the change is one constant pair + the invariant test stays green.
- **No automated match-length test exists** to measure this (confirmed: `scoring.test.ts` only asserts
  the `/3` ratio at :330-331), so the empirical check is the user's playtest, not CI.

**TESTS TO REWRITE (not delete — preserve the protection, inverted).**
- `combos.test.ts:52-63` (was "≥1 A→B differs from B→A") → assert the **8 pairs are now symmetric**
  AND **Triangle↔Circle is still asymmetric** (Wheel≠Star).
- `scoring.test.ts:126-132` (Line→Dot placeholder) → Line→Dot now = Filament (symmetric income).
- `vortex.test.ts:124+` (Spiral→Dot ≠ Vortex) → Spiral→Dot now = Vortex.
- `combos.test.ts:88-90` (Square→Dot / Circle→Line stay functional) → these are NOT in the 8 magic
  pairs (Square-Dot, Circle-Line are "none" pairs) → they correctly **stay functional**; keep as-is
  but re-verify against the new table.
- Keep `TABLE.size===36` (`combos.test.ts:19-21`), the N/14 count test (`comboCodexStore.test.ts:28-35`),
  and the `/3` invariant (`scoring.test.ts:330-331`) GREEN.
- Add: a new test asserting each of the 8 pairs yields identical `resultName` both orders, and that
  the discovered-count is unordered (forward+reverse of one pair = +1, not +2).

**SPEC/LOCKED AMENDMENT.** Update `src/combos.ts` header, `LOCKED_DECISIONS.md §6`, `SPARK_Blueprint.md
§V.1` to record: "8 one-way magic pairs are now order-SYMMETRIC (S98); Triangle↔Circle remains
order-dependent (Wheel/Star) by design; HUD counts unordered /14." This amends a LOCKED decision —
**covered by the user's explicit approval.**

**ROLLBACK.** Revert the combos.ts rows + test edits + doc edits; key format never changed so saves
are unaffected either way.

## P3 — Connection Preview (the new feature)

**OBJECTIVE.** While a player drags a primitive out of the spawn zone, render a **pulsating, faded**
preview of *exactly* the connection(s) it will form at the current position — so the player sees the
connect distance and the number of connectors before releasing. On release the real bond commits
(existing code) at full opacity in the player + shape color. Local-only, render-only, zero netcode
or sim changes.

**APPROACH.** New `DragPreviewRenderer` (Pixi `Graphics`, own layer) synced each frame from the main
render loop (`main.ts` near `avatarRenderer.sync(world, controls)` :1664 — precedent for passing
`controls`). Each frame, if `controls.state.kind === 'AttractDrag'`:
1. Resolve the dragged spark + type; compute **`targetRefPos = isClient ? controls.cursor : spark.pos`**
   — the *exact* expression `onUp` uses (`controls.ts:456`). **This is the fidelity crux.**
2. Evaluate the **release gates** (`computeReleaseGates` + `isInsideSpawnerZone` /
   `isInsideEnemyTerritory` / `MAX_RELEASE_REACH=120`). If the placement would be rejected, draw
   **no** preview (or a distinct "won't-commit" cue) so we never show phantom bonds.
3. Run the **pure exported pickers** (Pixi-free, take World+pos): `pickHostTargetPrimitive`
   (primary, `AUTO_BOND_RADIUS=60`), `collectHostMergeCandidates` (`MERGE_REACH_RADIUS=100`),
   `pickRedundantBondTargets` (≤2, angular-spread). Reproduce the merge component-dedup so the count
   matches the release exactly.
4. `g.clear()`, then for each would-be bond draw `drawBondVisual(g, {ax,ay,bx,by,
   visualEffectId: lookupCombo(draggedType, targetType).visualEffectId, colorA=colorB=player.color,
   alpha: PREVIEW_BASE_ALPHA * pulse, width, tick})`. Anchor `(ax,ay)` at the dragged spark's drawn
   position (≈ `targetRefPos`); `(bx,by)` at each target's `pos`. Player color =
   `world.players.get(localPlayerId).color` (placerColor convention; **not** ownerColor).
5. **Pulse** = sin oscillator on **wall-clock** (`performance.now()/1000`, so it breathes even while
   sim is paused mid-drag, e.g. during NONET): `alpha = clamp01(BASE + AMP*sin(t*HZ*2π))`, `HZ ≤ 1.0`
   (photosensitivity — matches hazardRing 0.8Hz / nonetCelebration 0.95Hz). Faded look: BASE≈0.4.

**LAYER / CONTRACTS.** Add `previewGraphics` to `app.stage` **after** StructureRenderer's layers /
SparkRenderer.container but **before** `aboveFogLayer` → above the world, below fog + HUD; inherits
screen-shake (good). **Do NOT** add to `aboveFogLayer` (would break the `fog.spec.ts` aboveFog
children-count contract). `g.clear()` every frame (drawBondVisual doesn't clear). Pixi v8 arc bug:
`moveTo` before any arc/curve. Mirror the removed S53 P2 `previewGraphics` construction/`destroy()`
pattern.

**DETERMINISM / HONESTY.** Connection logic has zero RNG → preview == release on host/solo (same
world, same `targetRefPos`). On the **joiner**, the local snapshot may lag the host's authoritative
re-pick (host re-derives primary+merge for remote-origin, but NOT redundancy) → the joiner preview is
**best-effort** (matches its own optimistic prediction). Acceptable + documented; the host/solo path
(the common case) is exact.

**TESTING.** Pure-logic unit tests for the new preview-resolution module (feed a World + position,
assert the returned target/merge/redundancy set equals what `onUp` would dispatch — reuse the same
pickers so this is a guard against drift). Gate-rejection case → empty preview. No Pixi in unit tests
(the resolver is Pixi-free; the renderer is a thin draw shim). Live-verify via preview tools: drag a
spark near 0/1/2/3 targets, confirm the preview lines match the bonds that form on release, confirm
color/pulse, confirm nothing renders when over spawn-zone/enemy-territory or beyond reach.

**ROLLBACK.** Remove the renderer construction + the one `sync` call + the resolver module; pure
additive, nothing else references it.

## BATCH-LEVEL RISKS
- **R1 (P1):** branch-protection required-check name. Mitigated by keeping job name `e2e` + verifying
  via `gh api`.
- **R2 (P2):** the 4 inverted invariant tests must assert the NEW truth, not just be deleted, or the
  symmetry protection is lost. Explicitly rewritten above.
- **R3 (P2):** N/14 must stay 14 (count distinct magics, not 22 keys) — discovery canonicalization.
- **R4 (P2 balance):** shipping with no win-score change could feel too fast in casual play — held as a
  one-line playtest-gated follow-up, not a blocker.
- **R5 (P3):** preview must use `targetRefPos` (spark.pos host/solo), not raw cursor, AND evaluate
  gates — else it lies. Both built into the approach.
- **R6 (P3):** per-frame O(N) pickers ×3 during drag — N (primitives) is small (tens), drag is a
  single-spark gesture → negligible; only runs while `AttractDrag`.
